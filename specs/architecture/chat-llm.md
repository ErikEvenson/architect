# Chat and LLM Integration Architecture

## Overview

The architecture chat system provides an LLM-powered design assistant that can search the knowledge library, create and modify artifacts (diagrams, documents), record architectural decisions, and manage discovery questions — all through a streaming chat interface.

## Components

### Chat API (`src/api/chat.py`)

- **Endpoint:** `POST /api/v1/chat`
- **Protocol:** Server-Sent Events (SSE) for streaming responses
- **Provider support:** Local (OpenAI-compatible, e.g., LM Studio) or Anthropic API
- **Max tokens:** 16384 (prevents truncation of long artifact generation)
- **Tool-call loop:** Up to 20 rounds of tool execution per request

### System Prompt Assembly (`src/services/chat_service.py`)

The system prompt is assembled dynamically per request:
1. **Role definition** — architecture design assistant behavioral rules
2. **Available tools** — knowledge search, artifact CRUD, ADR/question management
3. **CLAUDE.md content** — project instructions and constraints
4. **WORKFLOW.md** — architecture session workflow steps
5. **Project context** — current project/version summary (artifacts, ADRs, questions)

### RAG Context Pre-fetch (`src/services/chat_service.py`)

Before each LLM call, the user's latest message triggers a semantic search:
1. Search knowledge embeddings for the top 10 results (min score 0.35)
2. Load the full content of the top 2 matching knowledge files
3. Inject as a system message before the last user message
4. Instruct the LLM to base its response on knowledge library content

### Tool Execution

Tools available when a version is selected:
- `search_knowledge` — semantic search across knowledge library
- `read_knowledge_file` — read full content of a specific knowledge file
- `list_artifacts` / `read_artifact` / `create_artifact` / `update_artifact` — artifact CRUD
- `trigger_render` — render an artifact to produce output files
- `list_adrs` / `create_adr` — ADR management
- `list_questions` / `create_question` / `update_question` — question management

Tools available without a version:
- `search_knowledge` / `read_knowledge_file` — knowledge library only

### SSE Event Types

| Event Type | Payload | Description |
|---|---|---|
| `content` | `{delta: "text"}` | Streamed text chunk from LLM |
| `tool_call` | `{name, arguments, tool_call_id}` | LLM requesting tool execution |
| `tool_result` | `{name, result, tool_call_id}` | Result of tool execution |
| `error` | `{message}` | Error during streaming or tool execution |
| `done` | `{}` | Stream complete |

## Data Flow

```
User message
  → Frontend sends POST /chat with messages + provider config
  → Backend builds system prompt (CLAUDE.md + WORKFLOW.md + project context)
  → Backend pre-fetches RAG context (semantic search → top 2 knowledge files)
  → Backend calls LLM API with system prompt + RAG context + messages + tools
  → LLM streams response via SSE
  → If tool_call: execute tool → append result → call LLM again (loop)
  → If content: stream to frontend
  → Done event signals completion
```

## Provider Configuration

| Provider | Base URL | API Key | Notes |
|---|---|---|---|
| Local (LM Studio) | `http://localhost:1234/v1` | Not needed | OpenAI-compatible API |
| Anthropic | Anthropic API URL | Required | Via OpenAI SDK compatibility |

## Security Considerations

- API keys for LLM providers are sent per-request from the frontend (not stored server-side)
- Knowledge library content is injected into prompts — no customer data leaks to the knowledge search
- Tool execution is sandboxed to the current version's data
- Content generated by the LLM is stored as artifact source code (project data in DB, not in repo)
